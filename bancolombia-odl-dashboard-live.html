Arquitectura Propuesta — MongoDB Atlas ODL
Cómo MongoDB Atlas + Atlas Stream Processing resuelve cada punto de dolor identificado
Propuesta de valor: Reemplazar la capa CDC problemática con una extracción Journal-based (cero impacto al core), simplificar el stack eliminando Apache Flink y usar Atlas Stream Processing para ensamblaje y transformación en tiempo real. MongoDB Atlas como ODL unificado para todos los canales digitales con latencia <400ms y 99.999% disponibilidad.
Arquitectura del TFW — Lo que está corriendo ahora
Infraestructura real desplegada para validar la arquitectura propuesta · Todo en vivo
Sobre este TFW: Para validar la arquitectura propuesta sin acceso al AS/400 real, construimos un simulador que genera transacciones fragmentadas con la misma estructura y volumetría (~500-3,000 TPS). Los datos fluyen en tiempo real a través de Confluent Cloud Kafka hacia Atlas Stream Processing, que ensambla los fragmentos y escribe documentos limpios en MongoDB Atlas. Este dashboard consume directamente esos datos en vivo.

Dead Letter Queue — Salud del Pipeline
Caso de uso: Monitoreo de resiliencia del pipeline. Los fragmentos que no cumplen el schema de validación o llegan incompletos al tumbling window se clasifican por severidad (CRITICAL/HIGH/MEDIUM) y se enrutan a la DLQ para reprocesamiento.
// MongoDB Aggregation — DLQ por Severidad y Tipo de Error db.movimientos_dlq.aggregate([ { $group: { _id: { severity: "$error_classification.severity", type: "$error_classification.error_type" }, count: { $sum: 1 } }}, { $sort: { "_id.severity": 1 } } ]) // Índice: { "error_classification.severity": 1 }
Throughput en Tiempo Real

Transformación: iSeries Legacy → ODL MongoDB
Décadas de deuda técnica convertidas en un documento limpio por Atlas Stream Processing
Caso de uso: Visualización de la transformación core del pipeline. El AS/400 emite ~200 campos por fragmento con FILLERS, códigos de routing, timestamps de migración y campos COBOL. ASP transforma 3×200 campos legacy en ~30 campos ISO normalizados.

Distribución Horaria de Transacciones
Capacity planning — ¿cuándo se concentra la demanda de canales digitales?
Caso de uso: Identificar picos de demanda para dimensionar auto-scaling y planificar ventanas de mantenimiento. Permite anticipar la carga y ajustar recursos del cluster MongoDB por horario.
// MongoDB Aggregation — Distribución Horaria db.movimientos.aggregate([ { $group: { _id: { $hour: "$movimiento.fecha_iso" }, count: { $sum: 1 } }}, { $sort: { "_id": 1 } } ]) // Resultado: [{ _id: 8, count: 12340 }, { _id: 9, count: 18500 }, ...] // Analítica: COLLSCAN → En producción usar Atlas Charts

Latencia End-to-End — Producción → MongoDB
Cristian: "que no sea más de 1.5 segundos" · Medido en cada documento vía _asp_metadata
—
Metodología: Cada fragmento lleva un timestamp _produced_at desde el generador (simula CDC del AS/400). ASP calcula e2e_latency_ms = last_seen - produced_at_min (end-to-end completo) y assembly_latency_ms = last_seen - first_seen (convergencia de fragmentos en el window).
// MongoDB Aggregation — Estadísticas de Latencia E2E db.movimientos.aggregate([ { $group: { _id: null, e2e_avg: { $avg: "$_asp_metadata.e2e_latency_ms" }, e2e_max: { $max: "$_asp_metadata.e2e_latency_ms" }, assembly_avg: { $avg: "$_asp_metadata.assembly_latency_ms" }, docs_medidos: { $sum: 1 } }} ]) // e2e_latency_ms = last_seen - produced_at_min (producción → MongoDB) // assembly_latency_ms = last_seen - first_seen (convergencia en window)
